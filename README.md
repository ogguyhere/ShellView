# ShellView

A Windows x64 reverse engineering and static binary patching project demonstrating shellcode injection into a closed-source application without modifying any external file or source code. All malicious code resides entirely within the patched executable.

This project was developed as part of an Information Security assignment (BSCS23128-B, Semester 2025) covering PE structure analysis, code cave exploitation, and C2 integration via binary patching.

---

## Overview

The target application is **IrfanView (i_view64.exe)**, a closed-source x64 image viewer. The assignment required reverse engineering the binary, identifying a suitable injection point, expanding the PE file to accommodate shellcode, and patching the executable so that the shellcode executes transparently on launch while the application continues to function normally.

Two separate patched executables were produced:

**Part 1 — Proof of Concept (CalcPopup)**
A calculator popup shellcode generated with msfvenom is injected into the expanded `.text` section. This demonstrates successful code execution without crashing the application.

**Part 2 — C2 Integration (Havoc TCP Reverse Shell)**
A TCP reverse connection payload generated by the Havoc C2 framework is injected into a newly added `.scod` PE section. On launch, the patched binary establishes a connection back to the Havoc teamserver while the original application continues running normally.

**Bonus — Threaded C2 Execution**
An assembly stub is prepended to the Havoc shellcode that calls `CreateThread` (via `Kernel32.dll`) to execute the payload in a separate thread. This prevents the shellcode from blocking or terminating the main application thread, allowing IrfanView to remain fully operational throughout the C2 session.

## Methodology

### 1. PE Expansion via CFF Explorer

The original `i_view64.exe` lacked sufficient raw (physical) space in its `.text` section to accommodate the shellcode, despite virtual space being available. CFF Explorer was used to increase the raw size of the `.text` section for Part 1, and to add an entirely new section named `.scod` for Part 2 (required due to the Havoc payload being approximately 102.4 KB).

| Section | Before (Raw Size) | After (Raw Size)   |
|---------|-------------------|--------------------|
| .text   | 0019E200          | 0019390 (expanded) |
| .scod   | did not exist     | 00104A00 (new)     |

### 2. Injection Point

A jump instruction in the original executable at address `00007FF61E2E061D` was identified as the injection point. The instruction was patched to redirect execution to the shellcode residing in the expanded section (`00007FF6D3BB061D` for Part 1, `.scod` start for Part 2). The original jump destination is preserved and jumped to at the end of the shellcode sequence to restore normal control flow.

### 3. Stack State Preservation

Immediately before the shellcode runs, a prologue pushes all general-purpose and extended registers (`push r15` through `push rax`) and aligns the stack. A corresponding epilogue pops them in reverse order before returning. This ensures the application does not crash due to corrupted register state.

The prologue occupies addresses `00007FF6D3C2F1F7` to `00007FF6D3C2F215` in the Q1 patched binary.

### 4. Shellcode Injection

The shellcode bytes are written directly into the code cave (Q1) or `.scod` section (Q2) using x64dbg's patch functionality. For Part 1, the msfvenom CalcPopup payload is placed after the register-saving prologue. For Part 2, the Havoc TCP reverse shell payload occupies the `.scod` section.

### 5. Returning to Normal Execution

Before the shellcode's natural termination (which would kill the process), a `jmp` is inserted to redirect execution back to the original instruction that was hooked. This ensures IrfanView continues running normally after the shellcode completes.

For Part 2 (Bonus), the main thread never enters the shellcode directly. Instead, a custom stub calls `CreateThread` with the Havoc shellcode's start address (`00007FF6A0F8D063`) as the thread function, then immediately returns to the normal application flow at `00007FF6A0E30458`. The C2 session runs entirely in the background thread.

## Shellcodes

### Part 1 — CalcPopup (msfvenom)

Generated with msfvenom. Pops the Windows Calculator as a visible proof of execution. The shellcode resides in the expanded `.text` section of the patched binary.

### Part 2 — Havoc TCP Reverse Shell

Generated by Havoc Framework v0.7. Establishes an HTTP-based reverse connection to the C2 teamserver. The shellcode resides in the `.scod` section of the patched binary.

Listener configuration used:

```
Listeners {
    Http {
        Name          = "IrfanViews"
        Hosts         = [""]
        HostBind      = "0.0.0.0"
        HostRotation  = "round-robin"
        PortBind      = 8080
        PortConn      = 8080
        UserAgent     = "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
    }
}
```

### Bonus — CreateThread Stub

A hand-written x64 assembly stub prepended to the Havoc shellcode. It saves registers, calls `CreateThread` with a pointer to the Havoc payload, restores registers, and jumps back to the normal application entry point. This decouples the C2 session lifetime from the application's main thread entirely.

## Key Technical Notes

- All shellcode is embedded statically within the patched executable. No external DLLs, scripts, or files are called at runtime.
- The patched binary retains full application functionality throughout shellcode and C2 session execution.
- The `.scod` section was necessary for Part 2 because the Havoc payload (~102 KB) exceeded the available space in any existing section without risking corruption of adjacent data.
- Stack alignment to 16 bytes is enforced before any call instructions in the custom stub to satisfy the x64 ABI calling convention.
- The early `jmp` return before the shellcode's built-in exit routine is critical for Part 1: allowing the shellcode to reach its own termination sequence would invoke `ExitProcess`, killing IrfanView.
- For the bonus, using a separate thread entirely avoids this problem and decouples the C2 session lifetime from the application's main thread.
